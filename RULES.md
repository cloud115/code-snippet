# JavaScript 代码片段库 - 项目规则

## 项目概述

这是一个包含各种JavaScript代码片段的项目，主要实现了设计模式、数据处理、函数工具、算法等多种功能。项目结构清晰，每个功能独立成一个文件，便于学习和参考。

## 代码风格指南

### 命名规范
- **文件命名**：使用数字+功能名称的格式，如 `01.Observer.js`
- **变量和函数**：使用驼峰命名法（camelCase）
- **类名**：使用帕斯卡命名法（PascalCase）
- **常量**：使用全大写字母和下划线（UPPERCASE_WITH_UNDERSCORES）

### 代码格式
- **缩进**：使用2个空格（根据现有代码风格）
- **分号**：必须使用
- **引号**：使用单引号（根据现有代码风格）
- **行宽**：建议不超过120个字符

### 注释规范
- **文件头部**：添加文件功能描述
- **函数和方法**：添加JSDoc注释
- **复杂逻辑**：添加内联注释

## 文件命名规范

- **格式**：`{数字}.{功能名称}.js`
- **数字**：从01开始，顺序递增
- **功能名称**：使用帕斯卡命名法，如 `Observer`、`DeepClone`
- **扩展名**：使用 `.js` 扩展名

## 开发流程指南

### 分支管理
- **main**：主分支，用于发布稳定版本
- **feature/**：功能分支，用于开发新功能
- **bugfix/**：修复分支，用于修复bug

### 提交规范
- **提交信息格式**：`类型: 描述`
- **类型包括**：
  - feat：新功能
  - fix：修复
  - docs：文档
  - style：样式
  - refactor：重构
  - test：测试
  - chore：构建

### 版本管理
- 使用语义化版本控制（Semantic Versioning）
- 版本号格式：`MAJOR.MINOR.PATCH`
  - **MAJOR**：主版本号，当你做了不兼容的API修改时递增
  - **MINOR**：次版本号，当你添加了向后兼容的功能时递增
  - **PATCH**：补丁版本号，当你做了向后兼容的bug修复时递增

### 代码审查
- 所有代码变更建议经过代码审查
- 代码审查重点关注：功能正确性、代码质量、性能优化

## 测试策略

### 测试类型
- **单元测试**：测试单个函数和组件

### 测试工具
- 使用Jest作为测试框架
- 测试文件放在 `__tests__` 目录中

## 文档生成

### 文档工具
- 使用JSDoc作为API文档生成工具
- 运行 `npm run docs` 生成文档
- 生成的文档放在 `docs` 目录中

### 注释规范
- 使用JSDoc注释格式
- 为函数、类、参数添加适当的注释
- 注释应包含功能描述、参数说明和返回值说明

## 代码风格检查

### 检查工具
- 使用ESLint作为代码风格检查工具
- 运行 `npm run lint` 检查代码风格
- 配置文件：`.eslintrc.js`

### 检查规则
- 缩进：2个空格
- 引号：单引号
- 分号：必须使用
- 行尾风格：Unix风格
- 禁止未定义变量
- 警告未使用变量

## 改进建议

1. **01.Observer.js**：修改removeObserver方法，使用对象引用或唯一ID查找观察者，避免同名观察者被错误移除
2. **02.PubSub.js**：修改notify方法，传递最新的消息而不是所有消息，或者根据业务需求调整逻辑
3. **03.DeepClone.js**：取消对Symbol属性处理的注释，完善深拷贝功能
4. **10.Concurrency.js**：优化并发控制逻辑，处理可能的边界情况，如空数组、负数并发数等
5. **14.Flat.js**：考虑使用更简洁的实现方式，如使用reduce和concat的组合
